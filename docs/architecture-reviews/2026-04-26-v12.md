# Architecture review — 2026-04-26 (post-v0.12)

**Cycle:** post-v0.12 re-evaluation cycle (per `` §6).
**Cadence trigger:** end of v0.12 (editor + terminal + agent attach).
**Previous review:** [`2026-04-26-v06.md`](./2026-04-26-v06.md) (post-v0.6).

This cycle covers two phases' worth of dependency drift. The
post-v0.9 cadence beat was missed in the build plan, so this review
re-validates the active stack against everything shipped in v0.7
(sandbox tier 1), v0.8 (drift / budget caps / critic), v0.9
(scheduling + durable resumption), v0.10 (local provider path:
Ollama + MLX), v0.11 (memory), and v0.12 (editor + terminal +
agent attach). v0.13 (sidecar Chromium browser) starts immediately
after the human signs off on the action items below, so the
browser-related ADR gets the close attention this round.

## Method

For each load-bearing dependency: pull the current upstream state,
compare to what was shipped or pinned in v0.7–v0.12, classify as
**still current**, **refine**, or **supersede**. Sources cited
inline.

---

## Findings

### ADR-0001 — Tauri 2 — **still current**

Tauri 2.10.x continues to be the production line; the IPC redesign
(Raw Payloads, custom protocols, the `tauri::ipc::Channel` type) is
what we depend on for sub-100 ms streaming. v0.7 (sandbox manager
state), v0.8 (kill / approval / drift command surface), and v0.9
(schedules) all rode the existing `State<...>` and command-handler
shapes without surprises. v0.12 added a fourth out-of-band
notification stream (`terminal:data`) on top of the existing
`chat:chunk` / `run:*` / `lsp:*` set; the broadcast-channel
multiplexing in `BrainSupervisor` (added in the v0.12 LSP commit)
now backs all of them. The Windows IPC slowness flagged at the v0.6
review remains untested; we still have not exercised the Windows
build path. Risk #3 in `02-architecture.md` §12 stays alive into
v0.18.
([Tauri 2 release notes](https://v2.tauri.app/release/),
[2.0 stable announcement](https://v2.tauri.app/blog/tauri-20/))

No action.

### ADR-0002 — React 19 + shadcn/ui + Tailwind v4 + Vite — **still current**

Three new surfaces (memory dialog, editor pane, terminal pane) used
the existing primitives without churn. The lazy-loaded editor
chunk is ~980 KB gzipped on its own bundle; the entry chunk stayed
flat at ~120 KB gzipped. No drift.

No action.

### ADR-0003 — Monaco — **refine (LSP integration shape)**

v0.12 was Monaco's first contact phase. The mount (commit 1) lazy-
loaded via React.lazy and survived the editor cold-start budget
test (~750 ms M-series, NFR1 budget is 5 s on a slow laptop). The
ADR-0003 "re-evaluate against CodeMirror 6" question resolves: the
bundle hit fits because of lazy load, and the inline-completion
API surface (a first-class hook in Monaco, hand-rolled in
CodeMirror 6) is what makes commit 3's ghost-text wiring a thin
shim instead of a feature.

The interesting choice was the LSP integration. The canonical
package is `monaco-languageclient` v10.7.0 (Feb 2026) with full
LSP-message routing and adapters for Monaco's language API
([monaco-languageclient docs](https://github.com/TypeFox/monaco-languageclient)).
We rolled our own thinner bridge instead — `LspClient` (renderer)
+ `LspManager` (brain) + `attachLspBridge()` glue — for three
reasons:

- **Bundle size.** monaco-languageclient pulls in the full
  vscode-languageclient stack (~few-hundred-KB) on top of Monaco;
  our hand-rolled glue is a few KB.
- **Brain symmetry.** Language servers are subprocesses; we
  already supervise subprocesses (sandboxes, providers) in the
  brain. Putting LSPs there matches the topology and gives one
  shutdown path.
- **Scope match.** v0.12 ships diagnostics + completion + hover.
  The full LSP spec (signature help, references, rename,
  semantic tokens) plugs into the same protocol bridge when those
  features land; we don't need vscode-languageclient's breadth on
  day one.

The trade-off: we re-implement what changes when the LSP spec
moves. The spec is mature and slow-moving, so this risk is small.
The refinement note already lives in ADR-0003 (added in the v0.12
LSP commit); this review confirms the call.

No further action.

### ADR-0004 — Brain process model: Python sidecar — **still current**

v0.7–v0.12 added six new subsystems inside the brain (sandbox
manager, drift monitor + critic, scheduler loop, local providers,
memory store, LSP manager, terminal observer) without crossing the
crash-isolation boundary. The single-Python-process call held up.

No action.

### ADR-0005 — IPC: NDJSON + JSON-RPC 2.0 — **still current; surface grew significantly + multiplexing landed**

The wire shape held across **18 new methods** added in v0.7–v0.12
(sandbox tier surface in v0.7, schedules.* in v0.9, providers.delta
in v0.10, memory.* in v0.11, lsp.* + inline.suggest + terminal.*
in v0.12), plus **5 new notification streams** (`drift.*`,
`schedule.*`, `lsp.*`, `terminal.*`). None required protocol
changes — they're additional methods/notifications on the existing
transport, exactly the extensibility the ADR claimed.

The supervisor refactor in v0.12 commit 2 promoted the per-call
mutex to a multiplexing pattern (persistent reader, per-id oneshot
channels, broadcast channel for notifications). This is the future
iteration the ADR-0005 supervisor docstring already flagged; the
new shape lets out-of-band notifications (LSP, terminal) flow
while another request is in flight without lock contention. The
existing `call` and `call_streaming` API surface is preserved.

`02-architecture.md` §6 still reflects the post-v0.6 surface and
needs another refresh to track the v0.7–v0.12 method/notification
explosion.

Action: refresh `02-architecture.md` §6.

([Tauri sidecar docs](https://v2.tauri.app/develop/sidecar/),

No action.

### ADR-0007 — LangGraph 1.0 + Claude Agent SDK — **still current; SDK surface grew**

LangGraph 1.0 GA continues to hold; the v0.5 plan-approval
interrupt, the v0.6 sub-agent runs-as-runs pattern, and the v0.9
durable resumption (resume_unfinished_runs) all rode the
SqliteSaver checkpoint surface without trouble.

The Claude Agent SDK shipped meaningful additions since the v0.6
review's `>=0.1.4` pin
([changelog](https://github.com/anthropics/claude-agent-sdk-python/blob/main/CHANGELOG.md)):

- **OpenTelemetry support** via `pip install claude-agent-sdk[otel]`
  (current as of 0.1.68, April 2026). Distributed tracing with
  W3C trace context propagation now flows from the SDK through
  the bundled CLI subprocess. Lines up neatly with ADR-0017's
  v0.14 instrumentation plan — we should pin the OTEL feature
  when that phase opens rather than building parallel
  instrumentation.
- **`SessionStore` adapter protocol** (5-method interface,
  `InMemorySessionStore` reference). When sub-agent transcripts
  start growing past in-memory, this is the cleaner persistence
  seam than re-implementing one ourselves.
- **`skills` parameter on `ClaudeAgentOptions`.** Useful when we
  start exposing our own agent skills (the `.claude/skills/`
  directory pattern). Not load-bearing yet.
- **Trio compatibility restored** via sniffio-based dispatch.
  Doesn't affect us (we're asyncio) but shows the maintainer is
  responsive to ecosystem breakage.

None of these change the orchestration architecture. The
v0.6-recorded refinement (sub-agents-as-top-level-runs vs
subgraphs, take-over as renderer-side flow) stayed correct
through v0.7–v0.12.

Action: when v0.14 (observability) opens, evaluate `claude-agent-sdk[otel]`
as the OTEL entry point rather than instrumenting the SDK ourselves.

### ADR-0008 — SqliteSaver durable execution — **still current**

The 2026 LangGraph guidance has firmed up: PostgresSaver for
production, SqliteSaver for development / single-user / local
workflows
([persistence docs](https://docs.langchain.com/oss/python/langgraph/persistence)).
That's our exact niche. v0.9's resumption gauntlet (kill brain
mid-run, restart, watch the run pick up where it left off via the
checkpointer) confirmed the per-run-db pattern works. The "skip
SqliteSaver and go straight to PostgresSaver for production"
advice in 2026 LangGraph community materials is for high-concurrency
deployments — not us. No drift.

No action.

### ADR-0009 — Memory: Mem0 + LangGraph + project files — **refine (record the v0.11 implementation + lock the recall-engine choice)**

v0.11 shipped the access layer + JSON-RPC + UI without committing
to Mem0 vs LangMem. The shape behind the API is SQLite + plain-
text LIKE search. The post-v0.6 review action ("evaluate
LangMem head-to-head when v0.11 starts") was deliberately deferred
because the API contract makes the swap trivial later.

Time to lock the call. The 2026 head-to-head
([Mem0 vs LangMem comparison](https://dev.to/anajuliabit/mem0-vs-zep-vs-langmem-vs-memoclaw-ai-agent-memory-comparison-2026-1l1k),
[Atlan 2026 frameworks](https://atlan.com/know/best-ai-agent-memory-frameworks-2026/)):

- **Latency.** Mem0 reports 0.2 s p95 retrieval. LangMem reports
  59.82 s p95 search latency — explicitly flagged as "never use
  for synchronous retrieval, async only." This kills LangMem for
  our chat-loop integration; we want recall in the prompt-build
  path, which is synchronous.
- **Memory shape.** Mem0's three-tier scopes (user, session,
  agent) match what we already wired (`scope` enum on
  `MemoryEntry`). LangMem's flat key-value store would force a
  schema collapse.
- **Integration breadth.** Mem0 has Python + JS SDKs, LangGraph
  integration, and an OSS path. LangMem is Python-only and
  LangGraph-native — the ergonomics are tighter for our stack
  but the latency story is disqualifying.

Decision: Mem0 stays the recall-engine choice when we add semantic
recall on top of the v0.11 SQLite layer. ADR-0009 needs a v0.11
refinement note recording the shipped implementation (SQLite +
LIKE) and the locked-in Mem0 decision.

Action: add a *Refinement at v0.11 implementation* note to
ADR-0009.

### ADR-0011 — Sandbox tiers — **refine (v0.7 implementation + Apple Containerization is now live)**

v0.7 shipped Tier 1 with Docker + devcontainer + git worktree as
the implementation. The threat model held: DNS-level egress
allowlist matches the runaway-prevention OQ-2 framing in
`01-requirements.md` §10.1; targeted-attacker defence stays in
the going-public hardening list.

Apple Containerization moved from "we estimated Sept 2026" (the
ADR's original timeline, corrected to Sept 2025 in the v0.6 review)
to **production**. Apple's `apple/container` repo is active
([apple/container](https://github.com/apple/container),
[apple/containerization](https://github.com/apple/containerization)),
the framework runs each Linux container in its own lightweight VM
on macOS 26 Tahoe with hardware-level isolation and a
SandboxService actor for managed sandbox operations
([Anil Madhavapeddy notes](https://anil.recoil.org/notes/apple-containerisation),
[The New Stack comparison](https://thenewstack.io/apple-containers-on-macos-a-technical-comparison-with-docker/)).

This is the path we want for Tier 2 on Mac when v0.15 lands —
direct on Apple Containerization rather than via Lima — plus
Lima as the fallback for older macOS. Lima itself has matured
([Lima 2026 microvm sandboxing notes](https://emirb.github.io/blog/microvm-2026/)),
but with Apple Containerization shipped, the bridge story we
sketched at the v0.6 review is no longer the recommended path on
Apple Silicon.

Action: add a *Refinement at v0.7/v0.12 architecture review* note
to ADR-0011 capturing both the v0.7 Docker-tier-1 implementation
and the Apple-Containerization-as-tier-2 plan for Mac (Linux Tier
2 stays Firecracker per the ADR; Lima becomes the macOS-fallback
path for users on macOS < 26 or where Apple Containerization
isn't a fit).

### ADR-0012 — Provider abstraction — **refine (v0.10 local providers landed)**

v0.10 added Ollama + MLX behind the same `LlmProvider` Protocol.
The abstraction held: `stream_chat` is the single normalisation
point; capability profiles are declared at provider construction;
the registry surface is unchanged. Two provider-specific
implementation notes worth recording:

- **MLX:** the `mlx-lm` import is lazy. Non-Apple-Silicon hosts
  surface a clean `ChatErrorChunk` instead of an import-time
  crash, so the same packaged binary still works on Linux /
  Windows. Current `mlx-lm` requires macOS 26.2+ to take
  advantage of M5 Neural Accelerators
  ([MLX install docs](https://ml-explore.github.io/mlx/build/html/install.html));
  older macOS keeps working at lower performance.
- **Ollama tool-calling:** we ship Qwen3-Coder-Next as the local
  default. The ecosystem has churned through several tool-call
  format issues during 2026 — Unsloth fixed Qwen3-Coder
  tool-calling for llama.cpp / Ollama / LMStudio
  ([Qwen3-Coder local guide](https://unsloth.ai/docs/models/tutorials/qwen3-coder-how-to-run-locally)).
  Reliability is "medium" today; we mark this in
  `CapabilityProfile` and our orchestrator chooses Anthropic
  when high-reliability tool-use is required (agent runner path
  selection).
- **Default Ollama context window:** Ollama defaults to 4096 token
  context; Qwen3-Coder-Next supports up to 256 K. We need to set
  the context size explicitly when spawning the model — flagged
  for a v0.10 / v0.11 follow-up if not already done.

Action: add a *Refinement at v0.10 implementation* note to
ADR-0012 capturing the lazy-import pattern, tool-calling
reliability tier per provider, and the Ollama context-window
override.

### ADR-0017 — Observability — **still current; OTEL surface clarified**

OpenTelemetry GenAI semantic conventions remain experimental as of
March 2026
([semconv docs](https://opentelemetry.io/docs/specs/semconv/gen-ai/)).
The transition plan now explicitly calls for stable-version
parity before marking GenAI conventions stable, dual-emission via
`OTEL_SEMCONV_STABILITY_OPT_IN`. The `claude-agent-sdk[otel]`
extra (see ADR-0007 finding) is the cleanest entry point we'll
have when v0.14 opens — better than home-rolling
instrumentation. The ADR-0017 plan still tracks; instrumentation
lands in v0.14.

No action.

### ADR-0018 — Python sidecar packaging: PyInstaller — **still current**

Filed at the v0.6 review as the supersession for ADR-0006. The
2026 Tauri+FastAPI+PyInstaller reference pattern
([Substack writeup](https://aiechoes.substack.com/p/building-production-ready-desktop))
is the canonical shape. We have not yet started the packaging
phase; nothing has shaken the choice.

No action.

### Croniter — **still current; flagged**

v0.9 pinned `croniter` for cron-expression parsing. The
maintainer announced potential discontinuation in late 2024 over
EU Cyber Resilience Act concerns, but the project is still
actively maintained — current version 6.2.2 (March 2026)
([croniter PyPI](https://pypi.org/project/croniter/)). No
official replacement is recommended; community alternatives
(python-dateutil, etc.) don't cover the same surface. Worth
re-checking at v0.15 if the maintainer's CRA deadline (June 2026
phase) actually triggers a wind-down.

No action this cycle. Logged so v0.15 review opens with the
question on the table.
