# Architecture review — 2026-04-29 (post-v0.25)

**Cycle:** second v2 re-evaluation cycle (per `` §7).
**Cadence trigger:** end of v0.25 (memory surface + five-tier closure).
**Previous review:** [`2026-04-28-v22.md`](./2026-04-28-v22.md) — first v2
cycle, post-v0.22 (auth-backend split + Thalyn identity + first-run wizard).

This cycle covers the v2 baseline phases v0.23 (lead lifecycle + brain →
lead delegation), v0.24 (worker model routing with per-project overrides),
and v0.25 (five-tier memory closure: personal-memory recall, lead-loaded
`THALYN.md`, worker → lead project-memory writes, scope filter in the
inspector).

## Method

Inventoried the accepted ADRs, gave special attention to those flagged
for re-check in the v0.22 review (Mem0 CVE, cef-rs bundling, OTel GenAI
semconv, Anthropic Agent-SDK billing). Web-searched current upstream
state and fetched the relevant GitHub issues/PRs for the carry-over
items. Sources cited inline.

## Findings

### ADR-0001 — Tauri 2 — **still-current**

Latest tagged release is still **Tauri 2.10.3 (March 4)** — no 2.11
since the v0.22 review. The Rust core grew the lead-lifecycle / routing
/ thread-context wiring this cycle without touching the IPC envelope or
permission set. The Windows IPC perf concern that has carried since the
v0.6 review remains untested but no upstream movement forces a posture
shift. ([Tauri releases](https://github.com/tauri-apps/tauri/releases))

No action.

### ADR-0002 — React 19 + shadcn/ui + Tailwind v4 — **still-current**

The shadcn/ui upstream story is unchanged from v0.22: forwardRef-removed
primitives, `data-slot` for styling, `sonner` replacing the deprecated
`toast`. The v0.23–v0.25 components (lead-lifecycle wiring through the
existing inspector, the new memory scope-filter chips) were authored
without `forwardRef` and without a toast surface, so the upstream picture
keeps not biting us. The forward-ref-free authoring convention noted in
the v0.22 ADR-0013 addendum continues to apply.
([shadcn/ui changelog](https://ui.shadcn.com/docs/changelog))

No action.

### ADR-0003 — Monaco — **still-current**

Untouched in v0.23–v0.25. The native LSP-namespace concern from prior
reviews stays parked behind any future LSP-scope expansion.

No action.

### ADR-0004 / ADR-0018 — Python sidecar + PyInstaller — **still-current**

PyInstaller **6.20.0 (April 22, 2026)** is the current line we already
pin against. The brain grew a handful of new modules this cycle
(`lead_lifecycle`, `lead_delegation`, `routing_table`, `worker_router`,
`routing_intents`, the v0.25 thread-context personal-memory recall, the
`record_worker_project_memory_write` helper) without changing the
sidecar spawn / supervise / IPC shape.
([PyInstaller releases](https://github.com/pyinstaller/pyinstaller/releases))

No action.

### ADR-0005 — NDJSON + JSON-RPC 2.0 IPC — **still-current**

This cycle added `lead.spawn / pause / resume / archive / list`,
`routing.get / set / clear / list`, and refined `memory.search` with a
`scopes` filter — all on the existing envelope. Streaming `thread.send`
absorbed the lead-delegation hop without changing the chunk shapes.

No action.

### ADR-0007 — LangGraph 1.0 + Claude Agent SDK — **still-current**

LangGraph **1.1.10 (April 27)** is the latest tag in the 1.x line. The
1.1.x stream added type-safe `version="v2"` for `stream()` / `invoke()`
(unified `StreamPart` / `GraphOutput.value` / `.interrupts`), Pydantic
+ dataclass coercion, and fixed time-travel-with-interrupts in
subgraphs. All opt-in / backwards-compatible — our orchestration is on
the legacy shape and stays valid. The 1.0 charter's no-breaking-changes
promise holds. ([LangGraph releases](https://github.com/langchain-ai/langgraph/releases))

The Agent SDK side is its own picture — see ADR-0020 below.

No action on ADR-0007.

### ADR-0008 / ADR-0022 — durable execution + eternal-thread durability — **still-current**

ADR-0022 is unchanged since v0.22. The randomized-kill soak still
passes at the per-push budget; the going-public-grade 1800s mode stays
on the deferred-hardening list. v0.25's personal-memory recall threads
through `assemble_context` without touching the durability path —
context assembly reads, it doesn't write.

No action.

### ADR-0009 — Memory: Mem0 + LangGraph + project files — **promoted to Accepted (this cycle)**

Two pieces of news this cycle:

1. **The CVE-2026-0994 carry-over from v0.22 is closed.** Mem0 PR
   [#4326](https://github.com/mem0ai/mem0/pull/4326) merged
   2026-03-14 and relaxed the protobuf pin to `>=5.29.6,<7.0.0`, so
   the patched protobuf is now reachable from a Mem0 install. The
   security argument for deferring Mem0 longer is gone; the
   FTS5-first deferral that ADR-0009 documents now rests on
   storage-simplicity grounds alone.
   ([Mem0 issue #3963](https://github.com/mem0ai/mem0/issues/3963))
2. **ADR-0009 finalised at v0.25** (status flipped from
   "Accepted (provisional)" to "Accepted") with a five-tier
   ownership block and explicit write-path enforcement
   (`record_worker_project_memory_write`).

The next decision the ADR will face is whether to light Mem0 up
behind the existing `memory.search` surface now that the CVE is
out of the way; that's a v0.26+ scope call, not this review's.

No new action this cycle.

### ADR-0010 / ADR-0019 — browser engine: cef-rs — **still proposed**

Issue [#209](https://github.com/tauri-apps/cef-rs/issues/209) was
**closed** (linked PR #321 merged 2026-01-01), but the merge delivered
a `tests_shared` library + a ported `cefsimple` example + sandbox
support + macOS / Linux build polish — **not** the on-demand
distribution mechanism the issue asked for. The closure is
optimistic; effectively cef-rs still has no shared-binary distribution
or downloader story for downstream Tauri apps. ADR-0019 stays
**Proposed**; promotion remains gated on real bundling.
([cef-rs PR #321](https://github.com/tauri-apps/cef-rs/pull/321))

No action this cycle. Re-check at v0.28.

### ADR-0017 — OpenTelemetry GenAI + self-hosted Langfuse — **refine (carry forward)**

The OTel GenAI semconv is **still experimental** as of the latest

([OTel GenAI semconv](https://opentelemetry.io/docs/specs/semconv/gen-ai/))

The v0.22 addendum still applies. Re-check at v0.28.

### ADR-0020 — brain auth-backend split — **still proposed (carry forward)**

[Issue #559](https://github.com/anthropics/claude-agent-sdk-python/issues/559)
remains open; the SDK still requires API-key auth for programmatic
calls and explicitly does not bill against Pro/Max subscriptions, while
the bundled `claude` CLI continues to authenticate fine against the
user's subscription. Our v0.22 finding (subscription auth ≠
subscription billing — the user's choice is about credential source,
not billing target) stands without revision. The
`CLAUDE_CODE_OAUTH_TOKEN` path called out in the issue thread is the
viable workaround if the gap matters to a specific user; we don't
require it today.

ADR-0020 stays **Proposed** with the v0.22 addendum still applicable.
No new action this cycle.
