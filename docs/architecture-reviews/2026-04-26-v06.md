# Architecture review — 2026-04-26 (post-v0.6)

**Cycle:** second re-evaluation cycle (post-v0.6).
**Cadence trigger:** end of v0.6 (sub-agent lifecycle + take-over).
**Previous review:** [`2026-04-26.md`](./2026-04-26.md) (post-v0.3).

This cycle re-validates the active stack after v0.4 (LangGraph
orchestration), v0.5 (plan-approval interrupt), and v0.6 (sub-agent
lifecycle, run-tree, take-over, depth cap) shipped against it.
v0.7 (sandbox tier 1) starts immediately after the human signs off
on the action items below, so the sandbox-tier ADR gets the close
attention this round.

## Method

For each load-bearing dependency: pull current upstream state,
compare to what we shipped or pinned in v0.4–v0.6, classify as
**still current**, **refine**, or **supersede**. Sources cited
inline.

---

## Findings

### ADR-0001 — Tauri 2 — **still current**

Tauri 2.x is the production line; the IPC redesign in v2 (custom
protocols, raw requests, removal of mandatory string serialization)
is what we depend on for sub-100 ms streaming. Windows IPC is
still the slowest path in the matrix — community benchmarks
report ~5 ms on macOS vs ~200 ms on Windows for a 10 MB transfer
([discussion](https://github.com/orgs/tauri-apps/discussions/11915)) — which keeps
risk #3 in `02-architecture.md` §12 alive. We have not yet
exercised the Windows path, so no new evidence either way; the
risk persists into v0.18 polish.

No action.

### ADR-0002 — React 19 + shadcn/ui + Tailwind v4 + Vite — **still current**

The combination is fully stable in 2026; shadcn/ui ships components
for Tailwind v4 + React 19 by default
([shadcn Tailwind v4 docs](https://ui.shadcn.com/docs/tailwind-v4)). Our v0.6
work (sub-agent tiles, take-over banner, modal flows) used the
existing primitives without surprises. No drift.

### ADR-0003 — Monaco — **still current (deferred)**

Editor work doesn't land until v0.12; nothing in v0.4–v0.6 touched
the choice. Monaco's bundle-size hit (~5–10 MB uncompressed) vs
CodeMirror 6 (~300 KB core) is the ongoing trade-off — we keep
Monaco for its IDE feature set and inline-suggest API surface,
mitigated by lazy-load when v0.12 arrives. No action this cycle.

### ADR-0004 — Brain process model: Python sidecar — **still current**

Sub-agent isolation in v0.6 reaffirms the choice: sub-agent runs
get their own LangGraph thread + checkpoint db, but they all live
inside the single Python sidecar process. Crash-isolation against
the Rust core continues to hold. No action.

### ADR-0005 — IPC: NDJSON + JSON-RPC 2.0 — **still current; surface grew**

The wire shape held up across the v0.6 surface: `runs.tree`,
`runs.kill`, `run.approval_required` with a new `gateKind: "depth"`
payload, and `parentRunId` stamped on every `run.status`
notification. None of these required protocol changes — they're
just additional methods/notifications on the existing transport.
The IPC method table in `02-architecture.md` §6 is out of date
relative to what we actually implemented, and §6 needs the
post-v0.6 update; that's a doc edit, not an ADR change.

Action: refresh `02-architecture.md` §6.

### ADR-0006 — Python sidecar packaging: PyOxidizer — **SUPERSEDE**

The v0.3 review noted PyOxidizer was de-prioritized; the 2026
state is more pointed. The maintainer
([indygreg, project status thread](https://github.com/indygreg/PyOxidizer/issues/741))
has not meaningfully updated PyOxidizer since January 2023, and
the wider 2026 packaging guidance has shifted to PyInstaller,
Nuitka, or cx_Freeze for the bundled-executable space
([pydevtools handbook](https://pydevtools.com/handbook/explanation/how-do-i-ship-a-python-application-to-end-users/),
[augmentedmind comparison](https://www.augmentedmind.de/2021/05/16/distribute-python-applications/)).
The 2026 Tauri-with-Python-sidecar reference patterns
([Substack writeup](https://aiechoes.substack.com/p/building-production-ready-desktop),
[example repo](https://github.com/dieharders/example-tauri-v2-python-server-sidecar))
all use **PyInstaller**, not PyOxidizer.

We are not yet on the packaging phase — the brain is still spawned
via `uv run python -m thalyn_brain` — so we have not absorbed any
PyOxidizer-specific cost. Switching the recommended path before
v0.x packaging actually starts is the cheap moment.

Action: file ADR-0018 (PyInstaller; uv-managed venv as the simpler
path during early packaging spike), supersede ADR-0006.

### ADR-0007 — Orchestration: LangGraph 1.0 + Claude Agent SDK — **refine (sub-agent shape)**

LangGraph 1.0 has been generally available since October 2025
([changelog](https://changelog.langchain.com/announcements/langgraph-1-0-is-now-generally-available),
[LangChain post](https://blog.langchain.com/langchain-langgraph-1dot0/));
we are on the GA line. v0.4–v0.6 exercised plan / approval /
execute / respond and the SqliteSaver-per-run pattern at length;
the orchestration substrate held up.

One implementation detail deviated from the original sketch and is
worth recording. The original ADR-0007 said *"sub-agents are
LangGraph subgraphs."* The v0.6 implementation instead spawns each
sub-agent as a **separate top-level LangGraph run** with its own
`thread_id`, its own per-run SqliteSaver checkpoint db, its own
audit log file, and a `parent_run_id` link in the runs index.
LangGraph's idiomatic subgraph pattern is single-checkpointer +
`checkpoint_ns` namespace differentiation
([persistence docs](https://docs.langchain.com/oss/python/langgraph/persistence),
[graph composition](https://deepwiki.com/langchain-ai/langgraph/3.6-graph-composition-and-nested-graphs)),
which is the right call when sub-agents are pure decomposition. Our
sub-agents are first-class navigable runs in the runs index — the
user opens, kills, and takes over a sub-agent on its own terms —
so independent persistence (one db per run, archivable per run) is
the better fit. The decision in ADR-0007's **Decision** section is
unchanged; the refinement note captures the shape.

Sub-agent take-over also landed differently than the architecture
sketch suggested. §6.1 listed `run.takeover` as a brain JSON-RPC
method; v0.6 implemented take-over entirely on the renderer side
using `runs.get` to snapshot the sub-agent's title + plan + final
response into a fresh chat session's system prompt. No brain
endpoint was needed. The brain may grow `run.takeover` if richer
history (action-log replay, tool-output context) becomes useful,
but it's optional, not load-bearing.

Action: add a *Refinement at v0.6 implementation* note to ADR-0007
covering both deviations. No supersession.

### ADR-0008 — SqliteSaver durable execution — **still current**

Plan approval + resume crossed the SqliteSaver boundary cleanly;
the per-run-db pattern survives sub-agent fan-out (each sub-agent
gets its own db). The 2026 best-practice signal for high-concurrency
LangGraph deployments is PostgresSaver
([persistence docs](https://docs.langchain.com/oss/python/langgraph/persistence)),
but Thalyn is single-user-on-laptop, so the trade-off the ADR
documented still holds. No action.

### ADR-0009 — Memory: Mem0 + LangGraph + project files — **on schedule (re-evaluate at v0.11)**

Memory work doesn't land until v0.11. The 2026 landscape has
matured: **LangMem** has emerged as "the developer's choice for
LangGraph shops" — free, owned by the user, native to our
orchestration substrate
([Mem0 vs Zep vs LangMem comparison](https://dev.to/anajuliabit/mem0-vs-zep-vs-langmem-vs-memoclaw-ai-agent-memory-comparison-2026-1l1k),
[Atlan 2026 frameworks](https://atlan.com/know/best-ai-agent-memory-frameworks-2026/)).
Mem0 still has the broadest feature set + project-file integration
we wanted; Zep is the time-indexed-graph power tool we don't need
at our scope. The original ADR's "Mem0 vs Zep is a candidate for
the v0.6 architecture review" note resolves this cycle as: keep
Mem0 the default, but evaluate **LangMem head-to-head** when v0.11
starts. No ADR edit yet.

Action: revisit at v0.11 kickoff with a written compare-and-decide.
Logged here so the v0.11 phase opens with the question on the table.

### ADR-0010 — Browser: sidecar Chromium + CDP — **still current; tooling solidified**

Stagehand v3 is GA on a CDP-native architecture (no Playwright
dependency, modular driver system); Browser-Use remains the
Python-native alternative. Both are actively maintained
([Stagehand v3 announcement](https://www.browserbase.com/blog/stagehand-v3),
[Stagehand vs Browser-Use 2026](https://www.nxcode.io/resources/news/stagehand-vs-browser-use-vs-playwright-ai-browser-automation-2026)).
The ADR called Stagehand v3 the initial choice; that's still the
best fit for our Rust-driven CDP path. Browser work doesn't land
until v0.13. No action.

### ADR-0011 — Sandbox tiers — **refine (macOS timeline corrected)**

The original ADR estimated Apple Containerization in
"macOS 26 Tahoe (Sept 2026)." Tahoe **shipped September 15, 2025**,
and Apple announced both the Containerization framework and the
`container` CLI at WWDC 2025
([Wikipedia](https://en.wikipedia.org/wiki/MacOS_Tahoe),
[Anil Madhavapeddy notes](https://anil.recoil.org/notes/apple-containerisation)).
That changes the bridge story: Lima is no longer the
keep-us-going-until-Apple-ships option — Apple Containerization is
already available, and Lima itself can bridge to it
([Lima discussion](https://github.com/lima-vm/lima/discussions/3622)).
The four-tier model is unchanged; only the macOS Tier-2 timeline
is wrong. v0.7 lands Tier 1 imminently; v0.15 is when Tier 2
matters, and by then we should pick Apple Containerization
directly on Mac (with Lima as a transitional path for users on
older macOS, or for environments where Apple Containerization
isn't a fit).

Firecracker on macOS via Lima also exists for users who
specifically want Firecracker semantics
([Firecracker-Lima-VM repo](https://github.com/yashdiq/firecracker-lima-vm),
[KubeCon EU 2026 — Lima for AI agent sandboxing](https://emirb.github.io/blog/microvm-2026/)).

Action: add a *Refinement at v0.6 architecture review* note to
ADR-0011 capturing the corrected timeline. No supersession.

### ADR-0012 — Provider abstraction — **still current**

The v0.3 refinement note (narrower Rust trait, single
`stream_chat`, deferred `ToolCallNormalizer`) is what v0.4–v0.6 ran
on. Sub-agents pick up the parent's provider through the runner's
spawner closure; the abstraction made multi-tier provider routing
a non-issue. Local-provider work doesn't land until v0.10; **Qwen3
(specifically Qwen3.5-Coder)** is the 2026 local front-runner for
tool-calling reliability among local options
([Morph 2026 ranking](https://www.morphllm.com/best-ollama-models),
[Ollama tool-calling docs](https://docs.ollama.com/capabilities/tool-calling)).
The model name carried in the v0.10 build-plan entry says
"Qwen3-Coder-Next" — that name still tracks; if upstream moves to
"Qwen3.5-Coder" by v0.10 we update the docs then. No action this
cycle.

### ADR-0013 — Design system (OKLCH + Geist + Motion) — **still current**

OKLCH tokens + Tailwind v4 `@theme inline` survived two
visual-regression baseline updates and the v0.6 sub-agent UI
without contrast or token drift. WCAG-AA fixes in v0.5 (accent
darkened to match the contrast ratio) hold. No action.

### ADR-0014, 0015, 0016 — Documentation, commit hygiene, license — **still current**

The leakage scanner caught two more near-misses across v0.4–v0.6
(both real); MADR ADR + Mermaid + this review pattern is working;
MIT is unchanged pending the going-public re-evaluation. No
action.

### ADR-0017 — Observability — **still current**

OpenTelemetry GenAI semantic conventions remain experimental in
March 2026
([semconv docs](https://opentelemetry.io/docs/specs/semconv/gen-ai/)) —
unchanged from the post-v0.3 review. Major observability vendors
(Datadog, Grafana) have begun native support; Langfuse v3 is the
self-hosted recommendation
([Langfuse self-hosting docs](https://langfuse.com/self-hosting/deployment/docker-compose)).
The ADR-0017 plan still tracks; instrumentation lands in v0.14.
No action.
