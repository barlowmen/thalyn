## Findings

### ADR-0021 — Agent hierarchy — **load-bearing for v0.36, no refinement needed**

v0.36 lands the recursive-case half of ADR-0021: top-level lead spawn
already shipped in v0.23, and v0.36 extends the same primitive to
sub-leads — same `agent_records` row shape, `parent_agent_id` carries
the hierarchy, `scope_facet` names the slice. The depth cap (2 in v1)
is enforced at spawn time in `LeadLifecycle.spawn_sub_lead` by walking
`parent_agent_id` upward, with an explicit `override_depth_cap` knob
the caller only flips after the `gateKind: "depth"` approval resolves.
Memory isolation is enforced at the storage layer
(`MemoryStore.list_visible_for_agent` + `assert_writable_by`)
*plus* the column-keyed direct-DB filter — a focused test bypasses
the API to assert the in-DB layout still respects the rule.

The v0.36 implementation is fully consistent with the ADR's text;
the "Consequences" section's "extension is additive" claim held —
no breaking changes to the v0.23 lifecycle surface, the merge plan
extension was a single field added in v0.35 (`re_parent_sub_lead_ids`)
that v0.36 just exercises end-to-end.

**No refinement needed.** The ADR captured the recursive case at
draft time; v0.36 is the implementation lap.

### LangGraph — **steady, no upgrade pressure**

Quick smoke against `pip show langgraph` (current pin 0.3.x as of the
v0.20 build): no breaking-change drumbeat in the project's release
cadence, and the lead-graph / worker-graph shape we adopted (one
LangGraph thread per lead, per-run SqliteSaver per worker) hasn't
been deprecated or substantially redesigned. Carry the existing pin
forward; defer any version bump to the next `/dependency-review`.

The recent industry conversation about "swarms" and "graph of agents"
patterns (LangChain blog, OpenAI Swarms, Anthropic Constitutional AI
Agents) doesn't unseat ADR-0021 — those are mostly *worker-tier*
abstractions, not the persistent management hierarchy v2 builds. Our
lead/sub-lead split sits one tier above those patterns and would
*compose* with a swarm-style worker dispatch if v1.x ever wanted to
adopt one. The load-bearing model in ADR-0021 ("a lead is a lead is a
lead, only the parent differs") is the simplification that lets the
nesting stay tractable, and that's still the right call.

### ADR-0028 — Brain owns SQLite storage — **unchanged**

v0.36's storage extension (`memory_entries.agent_id` surfaced through
the Python dataclass + helpers) is purely additive; the column has
existed since migration 003. No new migration needed. The yoyo
runner / 7-migration chain still applies cleanly under the
in-process cache short-circuit. SQLite 3.46+ (the macOS bundled
version) is unchanged on the macOS side. No upgrade pressure.

### ADR-0024 — Project mobility + pluggable classifier — **promoted last cycle, working as designed**

ADR-0024 flipped Proposed → Accepted in commit `cbc00c0` (the very
last commit of v0.35). The v0.36 sub-lead re-parent path exercised
the merge plan's `re_parent_sub_lead_ids` field for the first time
end-to-end (`tests/test_project_merge_with_sub_leads.py`); the field
existed since v0.35 but had been dormant. Behaviour matches the ADR
posture — the sub-lead's `parent_agent_id` flips to the surviving
lead, the project_id flips with it, and memory visibility flips
because the helper consults the registry on every read.

No refinement needed. Lock-in for v1.x.

### ADR-0023 — Worker model routing — **unchanged**

Sub-leads dispatch workers the same way top-level leads do (the
worker router doesn't branch on parent kind, only on
`(task_tag, project_id)`). v0.36 didn't touch the routing path. The
"per-project override + global default" model still holds.

### Other dependencies (smoke pass)

- **Anthropic Python SDK** — pin unchanged since v0.33; no breaking
  release.
- **Tauri 2.x** — no v0.36 frontend / Rust touch beyond the
  IPC surface lead.spawn_sub_lead now ships. The Rust core's IPC
  client doesn't need a refresh — the dispatcher's existing
  `lead.*` registration covers the new method.
- **uv / ruff / mypy / pytest** — toolchain unchanged. No friction.

### Going-public-checklist gaps surfaced this cycle

- **Hash-chained memory audit log.** v0.36 surfaces `agent_id` on
  memory entries but doesn't change the audit-log shape; the v1
  per-run NDJSON log still doesn't sign or hash-chain. Already on
  the going-public list (per F7.6); no action this cycle.
- **Sub-lead overflow UI.** F2.3's depth-cap default ("user override
  required for depth 3+") is enforced at the lifecycle layer but the
  renderer hasn't surfaced the gate yet — when the override path
  fires, the renderer needs a "your lead wants its own sub-lead"
  approval card. Goes in the v0.38 polish list.
