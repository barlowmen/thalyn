# Architecture review — 2026-04-26 (post-v0.15)

**Cycle:** post-v0.15 re-evaluation cycle.
**Cadence trigger:** end of v0.15 (sandbox tier 2 + tier 3 scaffolding).
**Previous review:** [`2026-04-26-v12.md`](./2026-04-26-v12.md) (post-v0.12).

This cycle covers v0.13 (sidecar headed Chromium browser + agent
browser tool + observability panel + per-step capture + browser-pane
reference), v0.14 (OTel GenAI instrumentation + local Langfuse stack
+ user-supplied Sentry DSN + observability settings panel +
zero-egress test), and v0.15 (Tier 2 sandbox scaffolding +
Tier 3 cloud sandbox scaffolding + tier-escalation policy +
sandbox-tiers doc rewrite).

## Method

For each load-bearing dependency: pull current upstream state,
compare to what shipped or pinned in v0.13–v0.15, classify as
**still current**, **refine**, or **supersede**. Sources cited
inline.

## Findings

### ADR-0001 — Tauri 2 — **still current**

Tauri 2.10.3 is the production line (Mar 4, 2026 patch); no 2.11 cut yet, no IPC redesigns since the v0.12 review. The Raw Payloads + Channel surface we depend on for chat streaming, run lifecycle events, and the new browser observability events is unchanged. v0.13 added a fifth out-of-band notification stream (`browser:*` lifecycle in addition to `chat:chunk` / `run:*` / `lsp:*` / `terminal:data`) on top of the existing `BrainSupervisor` broadcast multiplexing — no new surface required. The Windows IPC perf risk we logged at the v0.6 review remains untested; Risk #3 in `02-architecture.md` §12 stays alive into v0.18. ([Tauri 2 release feed](https://v2.tauri.app/release/), [GitHub releases](https://github.com/tauri-apps/tauri/releases))

No action.

### ADR-0002 — React 19 + shadcn/ui + Tailwind v4 + Vite — **refine (a11y baseline)**

The shadcn / Tailwind v4 / React 19 combo is unchanged at the dep-resolution level — no breaking releases since v0.12. The post-v0.15 finding here is operational, not architectural: the v0.13 browser surface introduced a colour-contrast regression that **failed Storybook axe-playwright on every commit between `70e4cdf8` and `39a2caad`** (browser surface through sandbox-tiers doc) without surfacing it to the autonomous loop, because the `gh run watch --limit 1` command kept latching onto the previous run. The fix landed in `02fcf37` (`fix(ui): WCAG AA contrast on coloured Badge tones + error alert`).

Three follow-ups worth pinning down so this category of slip doesn't repeat:

- **Update `.claude/CLAUDE.md`'s CI-watch idiom.** Today it reads `gh run watch $(gh run list -R … --limit 1 --json databaseId --jq '.[0].databaseId')`, which races the new run getting queued. Replace with `gh run list ... --jq '.[0] | select(.headSha == "<sha>") | .databaseId'` so the watch only attaches to the run for the just-pushed sha — and surface a "no run found yet" error rather than silently watching an older run.
- **a11y-test the existing tone-colour Badge usages.** `provider-selector` and `provider-switcher` use the same `tone="success"` / `tone="warning"` shape that the browser surface tripped on. They have no a11y-tested stories today; without coverage, the same low-contrast bug could regress. Add stories for those next time either component is touched.
- **Promote axe-playwright to a project-local invariant, not just a CI gate.** The CLAUDE.md hard-rule already says local Mac gates ≠ Linux CI; the architecture-review note from v0.6 onward should add an explicit "run `pnpm exec test-storybook --url http://127.0.0.1:6006 --browsers chromium` locally for any UI commit before pushing."

No new ADR required; the engineering hygiene refinement lands in the next CLAUDE.md edit.

### ADR-0003 — Monaco — **refine (LSP namespace)**

Monaco 0.55+ shipped a native `lsp` namespace — first-party LSP support inside the editor — and moved the nested `languages.{css,html,json,typescript}` to top-level namespaces. AMD build deprecated. `monaco-languageclient` 10.7.0 (Feb 4, 2026) is the current TypeFox companion. Our v0.12-era hand-rolled `LspClient` + `LspManager` glue is unaffected today, but the new native shape is worth a closer look at v0.16 if we expand LSP scope (signature help, semantic tokens, formatter integration) — it might let us shed glue. ([Monaco changelog](https://github.com/microsoft/monaco-editor/blob/main/CHANGELOG.md), [monaco-languageclient](https://github.com/TypeFox/monaco-languageclient))

No immediate action; revisit during the next LSP commit.

### ADR-0004 — Python sidecar (and ADR-0018 PyInstaller packaging) — **still current**

The Python sidecar grew three new modules in this cycle (`browser`, `browser_cdp`, `browser_rpc`, `browser_tool`, `error_reporting`, `tracing`, `orchestration/escalation`) without any change to the spawn / supervise / IPC shape. `BrainSupervisor` in the Rust core continued to handle the brain as a single child process; the v0.13 Chromium sidecar is supervised separately by the new `BrowserManager`. Two-supervisor topology held up cleanly.

PyInstaller path (ADR-0018) hasn't been exercised in this cycle — packaging work didn't move.

No action.

### ADR-0005 — NDJSON + JSON-RPC 2.0 IPC — **still current**

The v0.13 commits added eight new methods (`browser.attach` / `browser.detach` / `browser.status` / `browser.navigate` / `browser.get_text` / `browser.click` / `browser.type` / `browser.screenshot`) plus three capture methods (`browser.set_capture_dir` / `browser.clear_capture_dir` / `browser.capture`); v0.14 added two observability secret commands surfaced via the same plumbing. None of them required schema changes to the JSON-RPC envelope. NDJSON framing held up against larger-than-typical payloads (base64 PNG screenshots in `browser.screenshot` / `browser.capture` results) without observable issue.

No action.

### ADR-0007 — LangGraph 1.0 + Claude Agent SDK — **still current**

LangGraph 1.1.9 is the production line; type-safe `stream`/`invoke` (`version="v2"`, Mar 10), interrupt time-travel fix in 1.1.7, replay-state propagation fix in 1.1.9, OTel instrumentation handler-type fix in 1.1.8 — all in our favour for the v0.14 OTel work. Claude Agent SDK is 0.1.68 (Apr 25); 0.1.60–0.1.68 added W3C trace context propagation, the `SessionStore` adapter protocol, `import_session_to_store()`, plus assorted server-tool primitives. None breaking; nothing we currently use deprecated. ([LangGraph releases](https://github.com/langchain-ai/langgraph/releases), [Claude Agent SDK PyPI](https://pypi.org/project/claude-agent-sdk/), [SDK CHANGELOG](https://github.com/anthropics/claude-agent-sdk-python/blob/main/CHANGELOG.md))

The v0.6-era refinement (sub-agents as independent LangGraph runs, take-over renderer-side) continues to hold; the v0.14 OTel work nests cleanly inside it because each `Runner.run` opens an `agent.run` span and child runs inherit context naturally.

No action.

### ADR-0008 — SqliteSaver — **still current**

Persistence docs unchanged: SqliteSaver is the recommended path for local-first / single-user; PostgresSaver only for high-concurrency production. Some 2026 community pieces argue "skip SqliteSaver in production" but that's the high-concurrency case, not ours. ([persistence docs](https://docs.langchain.com/oss/python/langgraph/persistence))

No action.

### ADR-0009 — Mem0 + LangGraph checkpoints — **refine (alternative noted)**

Mem0 remains the dominant standalone memory layer (~52k stars, ~14M downloads); no breaking-shape changes. New since the v0.12 review:

- **Hindsight** publishes 91.4 on LongMemEval (vs Mem0 49.0, Zep/Graphiti 63.8) — strongest-overall claim today, though new and unproven.
- **Letta** (ex-MemGPT) continues as the OS-style self-managing-memory option; **LangMem** is still latency-disqualified for synchronous-recall paths. ([2026 comparison](https://dev.to/anajuliabit/mem0-vs-zep-vs-langmem-vs-memoclaw-ai-agent-memory-comparison-2026-1l1k), [Vectorize](https://vectorize.io/articles/mem0-alternatives))

No immediate action — Mem0 is wired and works. Worth a 1-day spike during a future memory expansion to compare Mem0 vs Hindsight on our own data, but not this review's call.

### ADR-0010 — Browser sidecar over CDP — **still current (heavily refined this cycle)**

The pre-v0.13 spike retired Risk #1 and refined ADR-0010 four times during v0.13: the panel framing (real Chromium window primary, in-Tauri panel observability-only); discovery + lifecycle (use the user's installed Chromium, no bundling, no download); brain CDP transport choice (thin in-house client over `websockets`, not Stagehand v3 / browser-use / Playwright); the renderer surface (presentational `BrowserView` + connected wrapper for Storybook a11y); and per-step capture (auto on tool calls, per-run subdirectory, swallowed failures). The 2026-04-26 dep snapshot confirms nothing in the field invalidates the "thin in-house CDP client" call — Stagehand v3 GA'd on the CDP-native architecture (validating the same premise), browser-use moved to Python-first agent loops, Playwright stays the deterministic baseline. ([Browserbase v3 launch](https://www.browserbase.com/blog/stagehand-v3))

`chromiumoxide` is the obvious escape-hatch if we ever revisit (actively maintained, tokio-async, full CDP type coverage). Not needed today.

The v0.15.x take-over (window-raise) follow-up is still pending; that's the OS-API helper noted in the latest ADR-0010 refinement.

No new ADR required.

### ADR-0011 — Sandbox tiers — **refine (Apple Containerization shipped early)**

The original ADR pegged Apple Containerization at "September 2026 with macOS 26 Tahoe." It actually shipped earlier — `apple/container` and `apple/containerization` are both open-source and active; the framework is supported from macOS 26 Tahoe (basic ops on macOS 15 with networking limitations). VM-per-container, Virtualization.framework + Hypervisor.framework underneath. ([apple/container](https://github.com/apple/container), [apple/containerization](https://github.com/apple/containerization), [Tahoe release notes](https://developer.apple.com/documentation/macos-release-notes/macos-26-release-notes))

The v0.15 Tier 2 scaffolding chose Lima as the macOS bridge and noted the Apple Containerization upgrade path. With Tahoe shipped, the v0.15.x image-management commit can plausibly target Apple Containerization directly on macOS rather than going through Lima — that's a real decision, not an "Apple ships eventually" hand-wave any more.

**Recommended action:** before the v0.15.x microVM image-management commit lands, run a half-day spike on the Apple Containerization Swift package to compare its rootfs/networking surface against Lima's. If the spike finds Apple Containerization's API materially cleaner, file a refinement to ADR-0011 documenting "macOS Tier 2 is Apple Containerization, not Lima" and treat Lima as fallback for older macOS. If they're roughly comparable, keep Lima as the canon (more stable, longer track record on macOS 15).

Lima 2.1.1 is the current line (Apr 3, 2026); v2.0 expanded scope to AI workflows. Firecracker 1.15.1 is the current line (Apr 7, 2026); the kvm-clock fix on Linux ≥5.16 host + x86_64 guest means snapshots taken pre-fix need regeneration — relevant only when image management ships.

### ADR-0012 — Provider abstraction — **refine (local-model lead has shifted)**

The Anthropic adapter via the Claude Agent SDK works; OllamaProvider + MlxProvider land per the v0.10 implementation. The dep snapshot shows the local-model lead has moved:

- **Ollama 0.19** (preview, Mar 30, 2026) re-platformed Apple-Silicon inference onto MLX — Qwen3.5-35B-A3B on M5 Max gets +57% prefill / +93% decode over the previous backend, requires >32 GB unified memory. ([Ollama MLX blog](https://ollama.com/blog/mlx))
- **Qwen3.5** family (Qwen3.5-35B-A3B and Qwen3.5-Coder variants) is the local default the field is shifting to. Qwen3.5 4B scores 97.5% on local tool-calling evals — beating models 5× its size and putting "tool-use reliability: medium" too low for the new local default.

**Recommended action:** refine ADR-0012's `tool_use_reliability` mapping. The v0.10 capability profile recorded `Medium` for Qwen3-Coder-Next; for Qwen3.5 + properly-tuned chat templates the right value is closer to `High`. This is a small refinement note on ADR-0012 plus a `docs/local-models.md` update; no superseding ADR required. Worth combining with the next quarterly model-roster review.

### ADR-0013 — Design system (OKLCH + Geist + Motion) — **refine (axe-coverage gap)**

Token system held up cleanly across v0.13–v0.15: three new surfaces (browser panel, observability settings, sandbox tier badge in tile metadata) layered onto the same OKLCH + Geist + Motion primitives without churn. The lazy-loaded editor chunk stayed flat at ~980 KB gzipped; the entry chunk stayed at ~120 KB.

The colour-contrast finding above (v0.13 Badge tone variants on small text) is the architectural lesson here. The dark palette tokens themselves — `oklch(56% 0.15 80)` for warning at 56% L — are correct: the `--*-fg` sibling tokens exist precisely so colour-on-tone pairings clear AA. The mistake in the original Badge variants was using `text-{tone}` (the tone colour itself) instead of `text-foreground` for small-text labels on a tinted bg.

**Recommended action:** small refinement to ADR-0013 documenting the Badge convention — "tone tokens drive border + tinted bg, not foreground text on small labels; foreground stays at `--text` / `--text-muted` so contrast clears AA." Same convention should be applied as we add new tinted-bg components.

No action.

### ADR-0017 — OTel + Langfuse — **refine (semconv stability + Langfuse v3 endpoint)**

Two implementation refinements landed in v0.14 (the four-span shape, default-no-op exporter, the once-only TracerProvider gotcha, and the auto-init on first `get_tracer()` call). The dep snapshot adds a couple of small operational notes:

- **OTel GenAI semconv is still experimental.** Use `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` to lock in the latest names and stop emitting v1.36-era ones. Naming churn since the v0.12 review: `gen_ai.output.type` introduced; `gen_ai.openai.request.response_format` deprecated; cache-token attrs added (`gen_ai.usage.cache_read.input_tokens`, `gen_ai.usage.cache_creation.input_tokens`). ([OTel GenAI spec](https://opentelemetry.io/docs/specs/semconv/gen-ai/))
- **Langfuse 3 OTLP path** is `/api/public/otel` (introduced in v3.22.0). The `observability/docker-compose.yml` and the README we landed in v0.14 both reference this path, so we're already on the supported shape. Active issues to watch: indefinite-hang reports on `/api/public/otel/v1/traces` against v3.120.0 ([#9900](https://github.com/langfuse/langfuse/issues/9900)); recommend latest patch when upgrading. ([OTel integration](https://langfuse.com/integrations/native/opentelemetry))

**Recommended action:** small refinement note to ADR-0017 — "set `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` in production-style environments; update cache-token attribute names when we wire prompt-caching telemetry."

End of Aug 2026: tenants can populate Allow Lists + `EWSEnabled=True` to keep specific apps working past Oct 1. Oct 1, 2026: EWS off by default in Exchange Online; Outlook add-ins blocked. Apr 1, 2027: hard cutoff. ([Microsoft Learn](https://learn.microsoft.com/en-us/exchange/clients-and-mobile-in-exchange-online/deprecation-of-ews-exchange-online))
