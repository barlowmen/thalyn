This cycle covers v0.37 (information-flow drift / accuracy critic,
which itself produced ADR-0027 in-cycle without triggering a review)

## Findings

### ADR-0027 — Information-flow drift critic — **load-bearing for v0.37, accepted, working as designed**

ADR-0027 promoted to Accepted in commit `bf3a2c9` (the last v0.37
commit), generalising the ADR-0008 drift critic from the single
"action-log vs report" hop to *every* information-relay hop in the
EM hierarchy. v0.38 didn't touch the audit path — the hooks shipped
in v0.37 (`audit_info_flow(mode=…)`, the `chat:<turnId>` synthetic
run id for direct-chat audits, the heuristic-first / LLM-escalation
design) hold without refinement.

The wider field-of-view on "agent self-evaluation" sits in two
buckets: heuristic-first drift detection (our shape — cheap,
deterministic, escalates to an LLM judge only above a threshold)
and full LLM-as-judge per-turn evaluation (more accurate but blows
the NFR1 chat latency budget on every relayed turn). The shape
hasn't shifted since the previous cycle; carry ADR-0027 forward.

**No refinement needed.**

### ADR-0013 — Design system — **refined for the status-token split**

v0.38 surfaced a real WCAG 2.1 AA failure on the dark theme's
``text-destructive`` foreground use: the token shipped at 52% L,
tuned for white-on-red button contrast, and that same value as
foreground sat at ~3.6:1 against the page background. The fix split
the status family into surface tones (``--danger`` /
``--success`` / ``--warning`` at button-bg lightness) and
foreground-text tones (``--danger-text`` / ``--success-text`` /
``--warning-text`` at brighter lightness for 4.5:1 against the page
bg). Tailwind utility ``text-danger`` is wired via
``--color-danger`` in ``globals.css``; every ``text-destructive``
foreground call site swapped to it. ``bg-destructive`` (the button
variant) keeps the surface tone.

This is a refinement of ADR-0013, not a supersede — the design
system's *shape* (OKLCH tokens, dark-first, named status families)
is unchanged. The token table grew from one-tone-per-status to a
surface/text pair. An ADR refinement note should land alongside
v0.38, but the design intent in §11.4 of the requirements doc
stayed verbatim: the same status hues, the same calm-blue-violet
accent.

**Action.** Refine ADR-0013 to document the surface/text split. The
token table in `src/design/tokens.css` is the authoritative
expression; ADR-0013 captures the rationale.

### App icon — **shipped from spec, Tahoe `.icon` layered variants deferred**

The A3 gapped-T sigil concept locked in v0.32-era design work and
sat as a reference PNG for half the build until v0.38. The v0.38
ship translates the §4.2 geometry ratios and the §11.9 OKLCH tint
into a reproducible build: ``scripts/build-icon-assets.py`` renders
the canonical SVG, raster `.icns` (10 sizes), raster `.ico` (7
sizes), the Linux freedesktop hicolor PNG set, and the MSIX
``Square*Logo.png`` set from a single source.

What v0.38 *can't* produce on a non-Tahoe build machine is the
``.icon`` Tahoe-layered format — that needs Apple's Icon Composer
(Xcode 16+). The going-public checklist now carries this as a
binary-release gate. The bundled `.icns` covers macOS in the
meantime.

Side-by-side dock validation against Linear / Raycast / Cursor /
Claude Desktop / VS Code (per `docs/design/icon-direction.md` §4.4)
needs the same Tahoe machine. Tracked alongside the `.icon` ship.

**No ADR.** The icon direction is a design doc, not an architectural
decision — the build path is reproducible, the deferral is
mechanical (no Tahoe host), and the going-public checklist owns the
follow-up.

### NFR1 performance gating — **renderer path gated, desktop runtime confirmation deferred**

v0.38 added two Playwright perf specs gated against generous CI
thresholds: drawer-open animation (≤ 1 s vs the NFR1 ≤ 200 ms
headline) and lead-chat drawer cold-mount (≤ 1.5 s). The
multipliers are intentional — the Linux runner-via-Vite path is
meaningfully slower than the desktop-runtime path that the headline
NFR1 numbers measure, and the gate's job is catching structural
regressions, not proving the absolute number.

The headline numbers (≤ 200 ms drawer; < 3 s cold-start on Mac, <
5 s on Windows; < 100 ms streaming-token p95 on Mac; < 250 MB idle
resident memory) still need a measured pass on the actual hardware
tiers. The going-public checklist now tracks each one.

**No ADR shift.** NFR1's targets stand; the v2 plan never claimed
the renderer path could prove them.

### Integrated WCAG 2.1 AA — **gated end-to-end**

The Storybook a11y harness has covered every component story since
the v0.20 baseline (73 stories at v0.38), but the *integrated*
chat-first shell — chat + drawers + command palette composed at
the application level — was not axe-tested. v0.38 lands
`tests/visual/shell-a11y.spec.ts` covering both themes idle, plus
dark with the logs drawer / lead-chat drawer / command palette
open. The latent failure that triggered the status-token split
above came from this gate.

A note on the gate's plumbing: TypeScript syntax in a Playwright
``addInitScript`` string silently fails to parse — the existing
visual specs (`shell.spec.ts`, `editor-perf.spec.ts`) hit this and
have run without their Tauri stub installed since they were
written. The new a11y spec uses plain JavaScript on purpose. A
follow-up can rewrite the existing specs the same way, but that
shifts their snapshot baselines and belongs in its own commit.

**No ADR.** NFR8's WCAG 2.1 AA bar is the standing constraint;
v0.38 added the gate that proves it.

### git-cliff CHANGELOG generation — **bootstrapped, regex widened**

The v2 plan called for a CHANGELOG generated by ``git-cliff`` at
every tag, but the file itself had never been bootstrapped — the
config sat in `cliff.toml` and CI never produced output. v0.38
bootstraps `CHANGELOG.md` with the full v0.20 → v0.38 history.

The ``tag_pattern`` widened from
``v[0-9]+\.[0-9]+\.[0-9]+`` to
``v[0-9]+\.[0-9]+(\.[0-9]+)?`` so the historical
``vMAJOR.MINOR`` tags (v0.20 .. v0.37) get individual sections
instead of collapsing into one giant unreleased entry. Once 1.0
ships with three-part tags the same regex still matches.

**No ADR.** The release-engineering posture (Conventional Commits +
``git-cliff`` + signed tags at release time) is unchanged.

## Cycle handoff

This is the **final pre-1.0 architecture review.** The next
architecture review attaches to the **first 1.x phase** (the
going-public-checklist walk-through that gates `v1.0`), and the
post-1.0 cadence per the v2 plan drops to once per minor release.

The v2 build plan ends at this tag. The path to `v1.0` is:
1. Walk every item in `docs/going-public-checklist.md`.
2. For each completed item, file the ADR change or commit it requires.
3. When the checklist is empty, cut `v1.0` with a release tag,
   notarized binaries, SBOM, and signed audit logs.

`03-prompt-plan.md` §29 stand: user-supplied declarative
classifiers, reversible project merge with undo window, Apple
Containerization native, CEF native Wayland, voice output,
